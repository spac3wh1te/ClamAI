## 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 桌面壳 | Tauri 1.x (Rust) | 嵌入 WebView2，提供 IPC |
| 前端 | React 18 + TypeScript + Vite | SPA 单页应用 |
| UI 库 | Tailwind CSS + Radix UI | 样式与组件 |
| 后端 | Go 1.22+ | HTTP 代理服务 |
| 路由 | gorilla/mux | HTTP 路由 |
| 数据库 | SQLite（默认）/ PostgreSQL | GORM ORM + 部分遗留 Raw SQL（逐步迁移中） |
| 认证 | JWT (HS256) + bcrypt | 管理员/用户认证 |
| 安全 | AC 自动机 + LLM 语义分类 + 正则威胁规则 + 系统行为分析 | 内容安全检测 |

## 认证体系

```
┌──────────────────────────────────────────────────────────────┐
│                        认证流程                               │
│                                                              │
│  管理端口 (/api/v1/*):                                       │
│    1. adminAuthMiddleware 校验 JWT                           │
│    2. PC 模式 (host=127.0.0.1): localhost 请求免认证         │
│    3. Server 模式 (host=0.0.0.0): 所有请求必须带有效 JWT     │
│    4. /auth/status, /auth/login, /auth/setup 等路径免认证     │
│                                                              │
│  代理端口 (/{provider}/v1/*):                                │
│    1. authMiddleware 校验 ClamAI API Key                     │
│    2. 支持: JWT token / 静态 API Key (clam-sk-xxx)           │
│    3. X-Internal-Analysis 头: 内部调用免认证                  │
│    4. 验证通过后，handleTransparentProxy 替换为上游真实 Key    │
│                                                              │
│  Token 生命周期:                                             │
│    Access Token:  2 小时有效                                 │
│    Refresh Token: 30 天有效, 一次性使用                       │
│    JWT Secret:    存储在 DB admin_secrets 表, 持久化         │
└──────────────────────────────────────────────────────────────┘
```

## 数据库层架构

### 现状（混合模式）

数据库访问层处于 **从 Raw SQL 向 GORM 迁移的中间状态**：

| 访问方式 | 调用数 | 说明 |
|----------|--------|------|
| `db.Exec/Query/QueryRow`（`*sql.DB`） | 114 处 | 遗留 Raw SQL，含 SQLite 特定语法 |
| `gormDB.Raw/Exec`（GORM + Raw SQL） | 27 处 | 通过 GORM 连接但仍是原始 SQL |
| `gormDB.Find/First/Create/Save/...`（GORM ORM） | 83 处 | 真正的 ORM 调用，跨数据库兼容 |

### SQLite 特定语法问题（阻碍 PostgreSQL 兼容）

| SQLite 语法 | 出现次数 | 涉及文件 |
|-------------|---------|----------|
| `PRAGMA table_info(...)` | 6 | db_init.go, gorm_init.go |
| `INSERT OR REPLACE INTO` | 13 | vector.go, auth_middleware.go, security_config.go, db_init.go, config_api.go, auth_jwt.go, ratelimit.go |
| `INSERT OR IGNORE INTO` | 3 | vector.go, db_init.go |
| `datetime('now')` | 2 | vector.go |
| `datetime(timestamp,'localtime')` | 1+ | handlers_admin.go |
| `DATE(timestamp,'localtime')` | 3 | db_stats.go (gormDB.Raw) |
| `?` 占位符（vs `$1,$2...`） | ~100+ | 几乎所有 raw SQL 文件 |

### `db_adapter.go` 适配层

已提供 `dbInsertOrReplace`、`dbInsertOrIgnore`、`dbNow`、`makePlaceholders` 等适配函数，但大部分 raw SQL 调用点未使用这些函数，直接硬编码 SQLite 语法。

### GORM 模型

`models.go` 定义了 27 个 GORM 结构体，覆盖所有表。`gorm_init.go` 的 `AutoMigrate` 已能自动建表。`db_init.go` 中的 CREATE TABLE/ALTER TABLE 逻辑为遗留代码，与 AutoMigrate 并行存在。

## 已知问题修复记录

### Provider API Key 掩码问题（已修复）

**问题**：切换主题时，Provider 的 API Key 被覆写成缩略值（如 `sk-s...eknd`）。

**根因**：`handleGetConfig` 在返回配置时对 `providers.ApiKeys[].KeyValue` 做了掩码处理。前端切换主题时通过 `GET /config` → `PUT /config` 保存完整配置（包含已掩码的 key），导致数据库被覆写。

**修复**：从 `handleGetConfig` 中移除对 Provider API Keys 的掩码处理（保留对 Gateway API Key 的掩码）。`/api/v1/providers` 和 `/api/v1/config` 现均返回完整 key 值。

**影响范围**：此修复之前的主题切换操作会导致 provider key 被掩码覆盖，重新保存即可恢复。

### 安全分析消耗为 0（已修复）

**问题**：Dashboard 安全分析消耗面板始终显示 0。

**根因**：`dbGetSecurityTokenStats` 中 SQL 过滤条件为 `path = '/analysis/v1/chat/completions' OR path = '/security/semantic-check'`，但实际安全分析代码写入的 path 为 `/system-analysis/internal`，导致完全匹配不到。

**修复**：改用 `call_type = 'security'` 字段匹配，不再依赖硬编码路径。同时 `by_type` CASE ELSE 从 `'other'` 改为 `'security_analysis'`。

### 安全分析占比超过 100%（已修复）

**问题**：Dashboard 安全分析消耗占比显示 363.1% 等异常值。

**根因**：分母 `stats.total_tokens` 来自 usage API（仅 `call_type = 'client'`），分子为安全分析 token。安全分析消耗超过客户端消耗时占比超 100%。
