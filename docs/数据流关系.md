## 路由匹配流程

```
请求路径: /siliconflow/v1/chat/completions
                │
                ▼
  providerMatchMiddleware 遍历 DefaultProviderRoutes
                │
                ├─ 匹配 "/openai/v1"         → 不匹配
                ├─ 匹配 "/anthropic/v1"      → 不匹配
                ├─ 匹配 "/siliconflow/v1"    → ✓ 匹配！
                │
                ▼
  注入 context:
    - providerRouteSpec = {Name:"siliconflow", PathPrefix:"/siliconflow/v1",
                           UpstreamBase:"https://api.siliconflow.cn",
                           UpstreamRewrite:"/v1", AuthType:"bearer", ...}
    - providerName = "siliconflow"
                │
                ▼
  handleTransparentProxy:
     路径重写: "/siliconflow/v1/chat/completions"
            → 去掉前缀 → "/chat/completions"
            → 拼接 rewrite → "/v1/chat/completions"
     上游 URL: "https://api.siliconflow.cn" + "/v1/chat/completions"
     认证替换: "Bearer clam-sk-xxxx" → "Bearer sk-real-upstream-key"
     不转发 Accept-Encoding (让 Go http.Client 自动解压 gzip)
     透传请求体 → 透传响应体
     写入代理日志 (is_proxy_call=1, 含上游请求/响应头)
     设置 cw.upstreamProvider 供外层中间件读取
```

## 请求日志记录机制

每次代理调用产生 request_logs 记录，支持按 provider、model、client_ip、call_type、success 等维度搜索筛选。

前端调用日志页面中：
- **搜索**: 支持按 provider、model、client_ip、call_type、success 等字段组合筛选
- **类型列**: "直连"(emerald) / "代理"(blue) / "client"(来自客户端Key)
- **状态列**: "OK" / "ERR" / "BLOCKED"(orange, 安全拦截)
- **详情**: 代理记录显示上游请求头/响应头；安全分析记录显示安全分析请求/响应头
- **导出**: 支持导出为 CSV

> **api_key_used 字段**：存储完整原始 key 值（不再脱敏），用于支持 TOP10 授权密钥调用按用户归因。统计展示时通过 `maskAPIKeyForLog` 辅助函数处理显示。`call_type` 字段区分 `client`（客户端调用）和 `security`（安全分析内部调用）。

> **安全分析消耗占比**：Dashboard 安全分析消耗占比 = securityTokenStats.total_tokens / (客户端 total_tokens + securityTokenStats.total_tokens)，分母包含所有类型 token 确保不超过 100%。

## 系统行为分析数据流（会话防护）

系统行为分析与用户分析任务完全隔离，独立运行。采用两层威胁检测架构以节省算力：

```
系统行为分析配置 (system_analysis_config)
       │
       ▼
启动时 initSystemAnalysis()
  - 初始化系统分析任务（单条，created_by=__system__）
  - 启动 30 秒周期的调度器 startSystemAnalysisScheduler()
       │
       ▼
周期性调度 (每 30 秒检查)
  - 检查配置 enabled、model 是否就绪
  - 查询到期任务 (next_run_at <= now)
  - 执行 executeSystemAnalysisTask()
       │
       ▼
executeSystemAnalysisTask (两层检测):
  ┌─────────────────────────────────────┐
  │ Stage 1: 规则威胁评分（零 AI 成本）    │
  │ - 调用 scoreThreat(logs)              │
  │ - 检测：注入/编码/异常频率/时段/重复  │
  │ - 返回威胁分数 (0-100) + 命中规则    │
  └─────────────────────────────────────┘
       │
       ▼ 评分 ≥ 30？
  ┌────────────┐   ┌─────────────────────────────────────┐
  │    否      │   │ Stage 2: AI 深度分析               │
  │ 跳过 AI    │   │ - 仅对高威胁 key 调用 LLM          │
  │ 自动评分   │   │ - 会话链还原 + 攻击链分析          │
  └────────────┘   └─────────────────────────────────────┘
       │
       ▼
   1. 更新 system_analysis_tasks.result_*
   2. 写入 system_analysis_task_history
   3. 写入 system_analysis_key_results (含 threat_score/signals)
        │
        ▼
前端会话防护页面展示:
    - 扁平表格布局：时间|级别|评分|API Key|描述|日志数|详情
   - 4 张统计卡片：分析引擎状态、高危威胁数(来自自身 keyResults)、总会话数、综合风险
   - 可展开行：威胁信号、维度分析（6维）、AI 分析报告
   - 支持按严重级别筛选、按 API Key/摘要搜索
   - 每 10 秒自动刷新
```

> **节省算力**：攻击相对于业务并非常态，大多数日志为正常调用。Stage 1 自动过滤低威胁会话，仅在威胁评分 ≥ 30 时才调用 AI。
> **去重分析**：通过 LastLogID 跟踪已分析日志，新日志才过 Stage 1，无新增日志直接跳过。
```

> **注意**：`request_logs.api_key_used` 字段存储的是完整原始值（不再脱敏），用于支持 TOP10 授权密钥调用按用户归因。`dbGetCallerTop10` 通过 `api_key_used != ''` 自然排除安全分析内部调用（其 key 为空）。统计展示时可通过 `maskAPIKeyForLog` 辅助函数处理显示。`dbGetSecurityTokenStats` 使用 `call_type = 'security'` 过滤安全分析记录。

## AI 智能体安全数据流

### 环境安全检测

```
前端 AgentEnvironment 页面加载
        │
        ▼
   GET /api/v1/agent/list
   handleAgentList:
     - getAgentDirMap() 扫描 ~/.claude, ~/.cursor, ~/.codex 等 Agent 目录
     - 检查目录和配置文件是否存在
     - 按已检测优先 + 名称字母排序（稳定排序）
        │
        ▼
   返回 Agent 列表: [{name, dir_path, has_dir, config_files}]
        │
        │  用户点击"安全检测"
        ▼
   GET /api/v1/agent/check/{agent_name}
   handleAgentDeepCheck:
     - 目录结构检查（敏感文件、权限）
     - 配置文件检查（安全设置、API Key 暴露）
     - 写入 agent_env_checks / agent_config_checks 表
        │
        ▼
   返回检测结果: {checks: [{check_type, status, detail, severity}]}
        │
        ▼   前端持久化到 state[agent_name]

   失败/警告项可展开查看文件列表 + 处置建议
```

### 运行时安全 + 日志审计

```
前端触发日志扫描（AgentLogAudit 页面）
        │
        ▼
   GET /api/v1/agent/logs/parse?agent_name=xxx
   handleAgentLogParse:
     - 定位 Agent 日志目录
     - parseAgentMessages() 解析多格式日志（JSONL/JSON/Markdown）
     - 对每条消息应用威胁规则匹配
        │
        ▼
   威胁规则匹配 (agent_security.go):
     - agent_behavior: 行为异常（5条规则，如敏感操作/数据外泄/权限提升/系统修改/网络请求）
     - agent_injection: 威胁注入（1条规则，prompt注入/指令操纵）
     - 匹配成功 → 生成 AgentRuntimeEvent（含 severity/content/source_file）
        │
        ▼
   SHA256(agent_name|timestamp|content) 去重
     - 计算 contentHash，检查 agent_runtime_events 表是否已存在
     - 仅写入新事件（增量扫描，避免重复）
        │
        ▼
   全部事件（含匹配和未匹配）持久化到 agent_runtime_events 表
        │
        ▼
   前端查询: GET /api/v1/agent/runtime-events
   handleAgentRuntimeEvents:
     - 服务端筛选: agent_name / severity / event_type / search
     - 返回分页结果 + sev_map 统计（各严重级别计数）
        │
        ▼   运行时安全页面 (AgentRuntimeSecurity):
   持久化事件表格 + 筛选 + 分页
        │
        ▼   日志审计页面 (AgentLogAudit):
   查询参数持久化 URL + 服务端检索 + 增量扫描
   统计面板: 各级别事件数量
```

> **增量扫描机制**：每次扫描解析全部日志，但仅将新事件（contentHash 不重复）写入数据库，已存在的事件跳过。确保多次扫描不会产生重复记录。

## 字段提取数据流

透明代理不做格式转换，但需要从请求/响应 JSON 中提取特定字段用于安全审核和用量统计：

```
请求体 JSON
       │
       ▼
  extractTextsForSecurity(body, spec.Security)
       │
       │  按 SecurityExtraction 规则提取:
       │  - RequestContentPaths: ["messages[*].content"] → 用户输入文本
       │  - RequestSystemPaths:  ["system", "messages[role=system].content"] → 系统提示词
       │
       ▼
  securityMiddleware → 关键词/语义/威胁规则检测

  extractModelFromBody(body, spec.Usage)
       │
       │  按 UsageExtraction 规则提取:
       │  - ModelPath: "model" → "Qwen/Qwen2.5-72B-Instruct"
       │
       ▼
  requestTrackingMiddleware → 记录到 request_logs

  extractTokensFromBody(body, spec.Usage)
       │
       │  按 UsageExtraction 规则提取:
       │  - InputTokenPaths:  ["usage.prompt_tokens"] → 输入 token 数
       │  - OutputTokenPaths: ["usage.completion_tokens"] → 输出 token 数
       │
       ▼
  requestTrackingMiddleware → 记录到 request_logs
