# ClamAI 故障排查指南

## 一、Provider API Key 被覆盖成缩略值

### 症状
数据库 `providers` 表中 `api_key` 字段值变成类似 `sk-s...eknd` 的缩略形式。

### 原因
旧版本中 `handleGetConfig` 在返回配置时对 Provider API Keys 做了掩码处理，主题切换时前端通过 `GET /config` → `PUT /config` 保存了带掩码 key 的完整配置，导致数据库被覆写。

### 修复方法

**方法 1**：重新保存 Provider 配置
1. 进入「模型管理」页面
2. 找到对应的 Provider，点击编辑
3. 重新填入完整的 API Key
4. 保存

**方法 2**：通过 API 重新设置 key
```bash
curl -X PUT http://127.0.0.1:8081/api/v1/providers/{provider_name}/key \
  -H "Authorization: Bearer <your_jwt_token>" \
  -H "Content-Type: application/json" \
  -d '{"api_key":"your_full_api_key_here"}'
```

**方法 3**：更新数据库（需谨慎）
```sql
UPDATE providers SET api_key = 'your_full_api_key' WHERE provider_type = 'siliconflow';
```

### 预防
确保 ClamAI 版本为最新，旧版本用户切换主题不会导致此问题。

## 二、外部工具无法连接代理服务

### PC 模式下连接被拒绝

**检查**：
1. 确认 ClamAI 服务正在运行（桌面应用托盘图标）
2. 确认代理端口未被占用（默认 38080）
   ```bash
   netstat -an | findstr 38080
   ```
3. 确认防火墙允许该端口
   ```powershell
   netsh advfirewall firewall add rule name="ClamAI" dir=in action=allow localport=38080,38081 protocol=TCP
   ```

### Web 模式下无法连接

**检查**：
1. 确认服务以 `--host 0.0.0.0` 启动（而非默认的 `127.0.0.1`）
2. 确认服务器防火墙开放了对应端口
3. 确认使用正确的 API 地址（带 provider 前缀）

### API Key 认证失败

**检查**：
1. 确认使用的是 ClamAI API Key（`clam-sk-xxxx`），而非上游服务商 Key
2. 确认 Authorization 头格式正确：`Authorization: Bearer clam-sk-xxxx`
3. 确认该 API Key 状态为活跃（在管理界面查看）

## 三、安全告警/威胁检测异常

### 安全告警为空但确实有敏感请求

**可能原因**：
1. 安全功能未开启 → 检查「防护策略 → 安全配置」中是否启用
2. 请求未经过安全检测 → 确认请求经过了 ClamAI 代理（而非直接请求上游）
3. 检测模式为 `detect` 而非 `block` → 只会记录告警不会拦截

### 语义检测超时

**可能原因**：
- 语义检测依赖内部 LLM 调用，如果配置的分析模型不可用或超时，会导致请求响应慢
- 建议在「防护策略 → 高级策略」中配置可用的分析模型

### 威胁规则未生效

**可能原因**：
1. 规则未启用 → 检查「防护策略 → 高级策略 → 威胁规则」中规则的 `enabled` 状态
2. 规则的正则表达式有误 → 测试正则表达式语法

## 四、威胁挖掘（系统行为分析）无结果

### 页面显示"已停止"或"配置未就绪"

**检查**：
1. 确认系统分析已启用：「防护策略 → 高级策略 → 系统分析」中开启
2. 确认配置了有效的分析模型
3. 确认配置了有效的 API Key

### 分析任务执行但无历史记录

**可能原因**：
- 分析任务刚启动或未到达下次执行时间（默认 30 秒调度间隔）
- 查看调度器日志确认任务是否触发

## 五、日志无内容或显示异常

### 服务日志为空

**可能原因**：
1. 日志级别设置过高 → 在设置中降低日志级别到 `debug`
2. 日志文件被清空或轮转

### 调用日志显示 `api_key_used` 为缩略值

**这是正常现象**：`request_logs.api_key_used` 字段存储的是掩码值（用于日志展示），不影响统计准确性。

### 自动刷新影响日志查看

**解决**：日志页面的服务日志自动刷新功能默认关闭，需要时可点击刷新按钮手动开启。

## 六、性能问题

### 请求延迟高

**可能原因**：
1. 代理端口与上游端口网络延迟大 → 考虑将 ClamAI 部署在网络条件更好的位置
2. 安全检测耗时长 → 关闭不需要的安全检测功能
3. 限流触发 → 检查限流配置

### 内存占用高

**可能原因**：
1. 日志数据量过大 → 定期清理历史日志（DB 中 `request_logs` 表）
2. Provider 缓存未失效 → 调用 `/api/v1/providers/sync-all` 手动同步

---

## 七、数据库问题

### 忘记管理员密码

**解决**：删除 `clamai.db` 文件，重新启动应用进入设置向导（会丢失所有配置）

### 配置文件无法保存

**可能原因**：
1. 数据库文件权限问题 → 确认程序有写入权限
2. 磁盘空间不足

### 迁移到 PostgreSQL 后报错

**检查**：
1. 确认设置环境变量 `CLAMAI_DATABASE_TYPE=postgres`
2. 确认设置 `CLAMAI_DATABASE_URL` 为有效的 PostgreSQL 连接字符串（如 `host=localhost port=5432 user=clamai password=xxx dbname=clamai sslmode=disable`）
3. 确认目标数据库已创建（`CREATE DATABASE clamai;`）
4. GORM AutoMigrate 会自动建表，无需手动执行 DDL
5. **注意**：当前仍有部分 Raw SQL 使用 SQLite 特定语法（`INSERT OR REPLACE`、`datetime()`、`PRAGMA`），可能导致 PostgreSQL 下报错。完整兼容性正在迁移中

---

## 八、构建与部署问题

### `go build -tags server` 失败

**可能原因**：
1. 前端未构建 → 先执行 `npm run build` 并将 `dist/` 复制到 `server/frontend/dist/`
2. Go 版本过低 → 确认 Go 1.22+

### Tauri 桌面应用无法启动

**检查**：
1. 确认 WebView2 已安装（Windows 10/11 通常预装）
2. 确认显卡驱动已更新
3. 查看 Tauri 日志（应用目录下的 `logs/` 文件夹）

---

## 九、日志级别动态调整

无需重启服务，可通过管理界面或 API 实时调整：

```bash
# 设置为 debug 级别
curl -X PUT http://127.0.0.1:8081/api/v1/config \
  -H "Authorization: Bearer <token>" \
  -d '{"gateway":{"log_level":"debug"}}'

# 查看实时日志
tail -f clamai-service.log
```

## 十、安全建议

1. **生产环境务必使用 HTTPS**：通过 `--ssl` 启用 TLS
2. **不要使用默认端口**：修改默认代理/管理端口避免被扫描
3. **定期更换 API Key**：在管理界面定期更换 ClamAI API Key
4. **及时更新版本**：关注版本更新，修复已知问题
5. **合理配置限流**：避免单个 Key 耗尽配额
