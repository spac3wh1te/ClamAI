# ClamAI Service 代码结构

## 概述

ClamAI Service 是一个基于 Go 的 API 代理网关服务，采用 **Provider-native 透明代理架构**，按模型服务商原生协议透传转发，支持用户认证、安全审核、威胁检测、系统行为分析、流量统计等功能。

- **端口**：Proxy 端口（模型 API）默认 8080，Admin 端口（管理 API）默认 8081
- **模式**：`server` 模式（`--host 0.0.0.0`）提供完整 Web UI + API；默认模式为本地代理模式
- **数据库**：SQLite（`clamai.db`），支持用户、API Key、提供商、流量记录、安全告警、威胁规则、系统分析任务等数据持久化

## 中间件执行顺序

```
Proxy 中间件链：
corsMiddleware → providerMatchMiddleware → apiLoggingMiddleware → rateLimitMiddleware → securityMiddleware → requestTrackingMiddleware → authMiddleware

Admin 中间件链：
corsMiddleware → apiLoggingMiddleware → rateLimitMiddleware → adminAuthMiddleware → securityMiddleware → requestTrackingMiddleware → authMiddleware
```

## ProviderRouteSpec 配置结构

每个 provider 的路由映射和字段提取规则通过 `ProviderRouteSpec` 集中定义：

```go
type ProviderRouteSpec struct {
    Name            string             // "minimax", "siliconflow"
    PathPrefix      string             // "/minimax/v1" — 对外暴露的路径前缀
    UpstreamBase    string             // "https://api.minimax.chat" — 上游地址
    UpstreamRewrite string             // "/v1" — 路径重写规则
    AuthType        string             // "bearer" 或 "x-api-key"
    Security        SecurityExtraction // 安全审核字段提取规则
    Usage           UsageExtraction    // 用量统计字段提取规则
}
```

路径重写示例：
- 请求 `/minimax/v1/chat/completions` → 去掉前缀 `/chat/completions` → 拼接 rewrite `/v1/chat/completions` → 上游 `https://api.minimax.chat/v1/chat/completions`
- 请求 `/glm/v1/chat/completions` → 去掉前缀 `/chat/completions` → rewrite 为空 → 上游 `https://open.bigmodel.cn/api/paas/v4/chat/completions`

## 系统行为分析与用户分析任务的区别

| 维度 | 系统行为分析 | 用户分析任务（安全广场） |
|------|-------------|----------------------|
| 创建者 | `created_by='__system__'` | `created_by=<user_id>` |
| 任务表 | system_analysis_tasks | analysis_tasks |
| 历史表 | system_analysis_task_history | analysis_task_history |
| 配置 | system_analysis_config（集中管理） | 每任务独立配置 |
| 调度 | 系统启动时自动创建单条 periodic 任务，30秒检查调度 | startPeriodicTaskScheduler 管理用户周期任务 |
| 隔离 | 完全独立，created_by 不同，互不可见 | 按 userID 隔离，用户只能看自己的 |
| 高风险响应 | 自动写入 security_alerts | 不自动写告警 |
